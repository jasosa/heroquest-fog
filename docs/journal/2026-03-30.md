## Items Completed

### ISSUE-011/013/014/015 + FEAT-020 ‚Äî Trap system redesign (coupled batch)
**Branch:** merged into `main` via `fix/ISSUE-013-followup`
**Files changed:** 14 (559 insertions, 367 deletions)

The five tightly-coupled issues identified on 2026-03-28 were planned and implemented together. Summary of each:

**ISSUE-011 ‚Äî Wrong black shield rule text**
The inverted shield rule text in `TrapInteractionPopup` was corrected. Minor text fix, but high priority because it was actively misleading players.

**ISSUE-013 ‚Äî Popup flow redesign**
The original two-button flow (`Reveal Trap / Disarm Trap`) was replaced with a four-action design:
- **Spring** ‚Äî triggers the trap, shows a configurable effect message, and closes on dismiss
- **Reveal** ‚Äî informational only; shows the trap image and rules but changes no state (warning stays on board for later)
- **Disarm** ‚Äî session-only, with a confirmation step before executing
- **Dismiss** ‚Äî closes without any action

Already-sprung traps open a read-only popup on subsequent clicks.

**ISSUE-014 ‚Äî Fog reset clears trap session state**
`resetFog` now clears both `springedTraps` and `disarmedTraps` in addition to the fog set, so all trap states return to their pre-session defaults after a reset.

**ISSUE-015 ‚Äî Disarm does not mutate edit-mode data**
The previous `disarmTrap` directly deleted from `placed`, permanently removing the trap from the quest. Replaced with a session-only `disarmedTraps: Set<string>`. Disarmed traps are hidden in play mode via `getTrapRenderMode` but remain in `placed` for the DM's edit session.

**FEAT-020 ‚Äî Spring message configurable per placed trap**
`TrapConfigDialog` gained a `springMessage` textarea and a `removeAfterSpring` checkbox:
- `springMessage` replaces the old `trapNote` field; the text is shown when the hero springs the trap
- `removeAfterSpring` controls whether the trap token disappears from the board after being sprung (e.g. a pit trap stays; a boulder trap might be removed)
- `setTrapSpringConfig` replaces `setTrapNote` in `placementState.js`
- `TokenOverlay` hides sprung traps with `removeAfterSpring: true` in play mode

**Data model changes:**
- `PlacedPiece.trapNote` ‚Üí `PlacedPiece.springMessage`
- `PlacedPiece.removeAfterSpring: boolean` (new)
- `useGameState`: adds `springedTraps: Set<string>`, `disarmedTraps: Set<string>`, `springTrap(anchorKey)`, `disarmTrap(anchorKey)` (session-only), updated `resetFog`
- `navigationGuards.js`: `isSessionDirty` updated to include `springedTraps`

### ISSUE-012 ‚Äî Trap popup centered on board panel
**Branch:** `fix/ISSUE-012`

`TrapInteractionPopup` was previously positioned relative to the full viewport, placing it in the center of the screen regardless of where the board was. Fixed by using `position: absolute` inside the board panel container (`position: relative`) so the popup always overlays the board image specifically. Also cleaned up lingering merge conflict markers in `heroquest-fog.jsx` introduced during the FEAT-018 merge.

---

### ISSUE-013 followup ‚Äî Spring result shows trap image; disarm requires confirmation
**Branch:** `fix/ISSUE-013-followup`

Two refinements discovered during manual testing after the initial redesign:

1. The spring result phase now displays the trap's image alongside the effect message ‚Äî same visual treatment as the reveal phase. Previously the spring result was text-only.
2. Disarm now requires an explicit confirmation step: clicking Disarm shows a confirmation modal before executing, preventing accidental disarms. This was later amended by a hotfix (`cbe3984`) that fixed `springTrap` closing the popup before the `spring_result` phase could render.

Also: a duplicate `<TrapInteractionPopup>` render was found in `GameScreen` (rendered twice simultaneously). Removed the redundant render (`ae5c192`).

### ISSUE-006 ‚Äî Chest click always opens popup, never falls through to fog reveal
**Branch:** `fix/ISSUE-006`

Added `openedChestsRef` and a `shouldInterceptChestClick` guard in `handleCell` (mirroring the existing `shouldInterceptTrapClick` pattern). Clicking anywhere on a fog-revealed, un-opened chest cell now opens the chest result popup, even if the tap misses the chest image. Previously, a tap on the cell background would run the fog reveal path instead.

**Files changed:** `useGameState.js` (+9 lines), `useGameState.test.js` (+41 lines)

### ISSUE-007 ‚Äî Special monster notes accessible on mobile
**Branch:** `fix/ISSUE-007`

In play mode, tapping a special monster token now calls `onShowTooltip` with the `specialNote` ‚Äî the same mechanism used by note markers. Previously the note was only accessible via hover, making it completely inaccessible on touch devices.

**Files changed:** `TokenOverlay.jsx` (+4 lines), `TokenOverlay.component.test.jsx` (+51 lines)

### FEAT-021 ‚Äî Apply-to-all for trap type spring message
**Branch:** `feat/FEAT-021` (current branch ‚Äî not yet merged into `main`)

`TrapConfigDialog` now includes an "Apply to all [Trap Type] traps in this quest" checkbox. When checked, saving the spring message bulk-updates every placed instance of the same trap type in the current quest. This allows the DM to configure a single spring message once and propagate it to all pit traps, all boulder traps, etc., rather than opening each one individually.
