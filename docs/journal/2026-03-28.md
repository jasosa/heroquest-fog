## UX Audit

Ran a full UX review of the application using the `ux` subagent, focused on play mode coherence. The audit compared all interactive elements (traps, chests, search markers, secret door markers, note markers, special monsters) across three dimensions: how interactivity is signalled, what feedback is given after interaction, and whether interaction is reversible.

### Key findings

**Inconsistencies found:**
- Trap warning was the only interactive element with zero feedback — clicking it silently swapped the image. Every other element (chest, search, secret door) shows a result popup.
- Special monster notes are hover-only — completely inaccessible on mobile/tablet.
- Edit mode action buttons (pencil, star, warning icon) used three different HTML elements and sizes for the same concept.
- `RoomConfirmDialog` was the only dialog in the app without backdrop dismiss.
- Chest click that misses the image area falls through to fog reveal instead of opening the chest.
- Empty note markers silently did nothing on click in play mode.
- `HeroPlacementPopup` did not re-show after fog reset.

**Positive findings:**
- The shared hover tooltip system is consistent in style across all elements.
- Mode badge (green/gold) is persistent and legible.
- Fog reveal loop is clean.

### Backlog items created from audit

Ten new items (FEAT-015 through FEAT-019, ISSUE-005 through ISSUE-010) were added covering all findings. See `docs/planning/Backlog.md`.

## Pipeline and Workflow Changes

### CLAUDE.md additions

Two new sections added to `CLAUDE.md`:

**Permissions block:**
```
On a feature branch (feat/* or fix/*): proceed autonomously — read, write, edit,
and run any files within this project directory without asking for confirmation.

On main: follow the default permission behaviour — ask before modifying any file.
```

**Updated workflow (steps 4–13):**
- Step 4: Create a git branch (`feat/FEAT-XXX` or `fix/ISSUE-XXX`), branching off `main`.
- Step 10: If all tests pass, commit on the feature branch with the item ID as commit message.
- Step 11: Update status to `committed` — never `done`.
- Step 12: Never merge the feature branch into `main` — the user merges manually.
- Step 13: Switch back to `main` and loop to the next `not_started` item. Only `not_started` items are eligible; never `committed` or `in_progress`.

### Agent pipeline used

Each feature followed this pipeline:

```
architect (if complexity: high)
      ↓
    ux agent
      ↓
   planner
      ↓
  swe agent (worktree isolation)
      ↓
 commit + status update
```

The `swe` agent ran in an isolated git worktree (`isolation: "worktree"`), leaving changes uncommitted in the worktree directory. After confirming tests passed, changes were committed in the worktree and merged into the feature branch via `git merge --no-ff`.

### Observed worktree behaviour

The `swe` agent with `isolation: "worktree"` does not auto-commit. After it completes, changes are staged but uncommitted in the worktree path. The correct sequence is:
1. `cd` into the worktree path
2. `git add <files>` and `git commit`
3. Back in the main repo: `git merge worktree-branch --no-ff`

## Features Implemented

### FEAT-015 — Trap Interaction Overhaul
**Branch:** `feat/FEAT-015`
**Tests:** 408 passing (baseline was 349 — +59 new tests)

Replaced the silent trap reveal (click warning → image swaps) with a full interaction popup. Heroes now have agency over how they interact with traps.

**New components:**
- `TrapInteractionPopup` — 4-phase popup: `options` → `revealed` → `confirming_disarm` → `disarmed`
- `TrapConfigDialog` — edit mode dialog for DM to set a custom note per placed trap

**State added to `useGameState`:**
- `pendingTrapInteraction: { anchorKey, isRevealed } | null`
- `pendingTrapConfig: { anchorKey } | null`
- `disarmTrap(anchorKey)` — removes piece from `placed` *(see follow-ups below)*
- `revealTrap(anchorKey)` — adds to `revealedTraps`, called by popup
- `openTrapInteraction`, `closeTrapInteraction`, `openTrapConfig`, `saveTrapConfig`

**Data model changes:**
- `PlacedPiece` gains optional `trapNote: string`
- `pieces.js`: each trap type gains a `trapRules: string` field
- `placementState.js`: new `setTrapNote(placed, anchorKey, trapNote)` pure function

**Visual:** Red/orange glow (`drop-shadow(0 0 4px #c0392b) drop-shadow(0 0 8px #e74c3caa)`) on trap warning markers — distinct from the amber/gold chest glow. Signals danger vs. treasure.

**Also included:** `TokenOverlay` updated so revealed traps in play mode are also clickable (opening the popup for late disarm).

### FEAT-018 — Navigation and Mode-Switch Warnings
**Branch:** `feat/FEAT-018`
**Tests:** 361 passing (baseline 349 — +12 new tests)

Two safeguard warnings added to `GameScreen` in `heroquest-fog.jsx`:

**Warning 1 — Play → Edit mid-session:**
Fires when the DM switches from play to edit mode and the session is dirty (fog revealed, chests opened, traps triggered, searches done). Shows an informational modal: "Session State Carries Over — opened chests, revealed traps, search counts, and fog will NOT be reset." Single "Continue to Edit" button. Backdrop tap also proceeds.

**Warning 2 — Back to Library with unsaved changes:**
Fires when the DM navigates to the library while in edit mode with unsaved placed/door/marker changes. Shows a destructive-action confirm: "Unsaved changes will be lost — go back anyway?" with "Stay Here" / "Go Back" buttons. Backdrop tap intentionally does nothing (no accidental navigation).

**New module:** `src/navigationGuards.js`
- `isSessionDirty(fog, openedChests, revealedTraps, revealedSecretDoors, searchedCounts)`
- `stableStringify(obj)` — key-sorted JSON for reliable snapshot comparison
- `hasUnsavedChanges(snapshot, current)` — compares serialised edit state against last-saved snapshot

Snapshot ref (`savedStateRef`) is initialised at `GameScreen` mount from quest data and updated on each successful save.

## Follow-ups Detected (FEAT-015 testing)

After switching to `feat/FEAT-015` for manual testing, several issues were found with the trap popup implementation. All added to `Backlog.md`:

| ID | Title | Priority |
|---|---|---|
| ISSUE-011 | Wrong black shield rule text (should say "fails the jump") | high |
| ISSUE-012 | Popup not centered on board image (uses full viewport) | medium |
| ISSUE-013 | Popup flow redesign: Spring / Reveal / Disarm / Dismiss buttons | high |
| ISSUE-014 | Reset Fog must reset all trap session state | high |
| ISSUE-015 | Disarm in play mode must not mutate edit-mode placed data | high |
| FEAT-020 | Spring trap: configure effect message + remove/keep per trap in edit mode | high |
| FEAT-021 | Default trap rules text should be editable per trap type | medium |

### Critical design issues found

**ISSUE-013 — Popup flow redesign:**
The intended button set is **Spring / Reveal / Disarm / Dismiss**, not the current "Reveal Trap / Disarm Trap". The semantics differ:
- **Reveal** should be purely informational — no state change, warning stays on board
- **Spring** is the action that actually triggers the trap (with configurable effect from FEAT-020)

**ISSUE-015 — Disarm mutates edit data:**
The current `disarmTrap` implementation deletes directly from `placed`, which is the shared edit-mode data. Disarming a trap in play permanently removes it from the quest. The fix (ISSUE-015) is to introduce a session-only `disarmedTraps: Set<string>` that hides pieces in play mode without touching `placed`. On fog reset, `disarmedTraps` is cleared and traps reappear.

**ISSUE-014 — Fog reset incomplete:**
Related to ISSUE-015 — once `disarmedTraps` and `springedTraps` (from FEAT-020) exist, `resetFog` must clear them all.

These four items (ISSUE-013, ISSUE-014, ISSUE-015, FEAT-020) are tightly coupled and should be planned and implemented together.
